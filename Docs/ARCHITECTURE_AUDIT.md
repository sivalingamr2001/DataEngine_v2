# ARCHITECTURE_AUDIT.md — DataEngine Solution

## Executive Summary

DataEngine is a .NET 10 solution split into four projects — `DataEngine.API`, `DataEngine.ReaderService`, `DataEngine.TransactionService`, and `DataEngine.Core` — that together expose a generic, metadata-driven read/write layer over MySQL and (partially) Oracle.

The read path (`FetchConfig` → `IGetDataService` → `QueryRepository`) is parameterized correctly and is the most solid part of the system. The write path (`TransactionService`) is **not production-safe**: table and column identifiers supplied by the caller are interpolated into SQL strings with no allow-list check, there is no idempotency protection despite a `TransactionId` field that implies one, there is no audit trail, and the `IValidationService` that exists in `DataEngine.Core` is never wired up or called. Oracle write support does not exist, and Oracle read support (`OracleGetDataService`) is an unimplemented stub. Filtering, sorting, and search fields on `FetchConfig` are accepted by the API but silently ignored by the only working implementation (`MySqlGetDataService`).

This is a working proof of concept for a single-database, trusted-caller scenario. It is **not** ready for a high-volume, multi-tenant, multi-database enterprise deployment without the fixes in `CRITICAL_ISSUES.md`.

## Current Architecture

```
Client
  │  HTTP JSON
  ▼
DataEngine.API  (Controllers: DataController, TransactionController)
  │
  ├──► DataEngine.ReaderService
  │       FetchConfig ─► IGetDataService (MySqlGetDataService | OracleGetDataService[stub])
  │                         ├─ SqlGuardian (regex blocklist firewall)
  │                         ├─ QueryRepository (stored query lookup, parameterized)
  │                         └─ DatabaseConnectionFactory (Core)
  │
  └──► DataEngine.TransactionService
          TransactionRequest ─► TransactionService
                                   ├─ FieldMapperRepository (schema metadata, parameterized)
                                   ├─ DatabaseConnectionFactory (Core)
                                   └─ raw SQL string built from caller-supplied
                                      table/column names (NOT validated)

DataEngine.Core
  ├─ DatabaseConnectionFactory  (primary + round-robin read replica, retry loop)
  ├─ DatabaseConnectionVerifier (startup ping)
  ├─ ValidationService / IValidationService  (built, but never registered or called)
  └─ DataTypeConverter (string → typed value coercion)
```

Notable structural issue: `DataEngine.Core` declares its own domain types (`DatabaseConfig`, `AuditOperation`, `DatabaseProvider`) under the `DataEngine.ReaderService.Domain` / `DataEngine.ReaderService.Enums` namespaces instead of a `DataEngine.Core` namespace. This means Core — which should be the shared foundation — is namespaced as if it were a sub-module of ReaderService, and `DatabaseProvider` is defined **twice** (once in `DataEngine.Core/Domain/DatabaseConfig.cs`, once in `DataEngine.Core/Enums/Enums.cs`), in two different namespaces, with identical members. This is a layering violation that will cause confusion (which `DatabaseProvider` did you reference?) and will make it harder to add new projects that depend on Core without also pulling in ReaderService naming.

## Strengths

- Clear separation of read (`ReaderService`) vs write (`TransactionService`) concerns at the project level.
- `QueryRepository` and `FieldMapperRepository` — the two repositories that read internal metadata tables — use parameterized ADO.NET commands correctly.
- `DatabaseConnectionFactory` implements a real retry-with-backoff loop and structured logging around connection acquisition, and separates primary vs. read-replica routing.
- Serilog + `UseSerilogRequestLogging` gives baseline structured request logging out of the box.
- `SqlGuardian` shows real intent to stop free-text SQL abuse for the direct-query path, with length limits, complexity limits (JOIN/subquery/UNION counts), and a broad pattern list.
- Metadata-driven `FieldMapper` design (field → column, default value tokens, `AutoGenerated`/`AllowUpdate` flags) is a reasonable foundation for a generic CRUD engine, and recursive child-record processing (`RenProps`) with a depth ceiling is a sound idea for parent/child payloads.
- Startup connection verification (`DatabaseConnectionVerifier`) fails fast instead of accepting traffic against a dead database.

## Weaknesses

- **No allow-list validation of table/column identifiers before they are string-interpolated into SQL** (`TransactionService.ProcessInsertAsync/ProcessUpdateAsync/ProcessDeleteOperationsAsync`, and the child/delete table keys in `TransactionRequest.RenProps`/`DelProps`). This is the single biggest structural problem in the solution — see `CRITICAL_ISSUES.md` P0-1.
- `SqlGuardian` is a **blocklist**, not an allow-list. Blocklists are inherently incomplete; new bypass techniques (encoding tricks, provider-specific functions, comment obfuscation) will eventually get through. It should validate against a known-safe grammar/allow-list of tables and columns instead of (or in addition to) pattern matching.
- `IValidationService` exists, has a real interface and a real (if MySQL-hardcoded-looking) implementation, but is **not registered in DI and not called anywhere** in `TransactionService.TransactionAsync`. Business-rule and schema validation is effectively dead code.
- No idempotency store: `TransactionId` is accepted from the caller and logged, but never checked against prior executions. A network retry (client timeout + resend) will insert the row twice.
- No audit table. `AuditOperation` enum exists in `Enums.cs` but nothing ever constructs or persists an audit record — no before/after image, no correlation ID, no IP address capture.
- `OracleGetDataService.ExecuteAsync` throws `NotImplementedException` after opening a connection — so Oracle is configured as if fully supported (`DatabaseProvider.Oracle` accepted in `Program.cs`, `DatabaseConnectionFactory`, DI registration) but will fail every read request at runtime. There is no Oracle write path at all.
- `FetchConfig.EnableServerSideFiltering`, `FilterConditions`, `EnableServerSideSorting`, `SortField`, `SortDirection`, and `SearchText` are all present on the request contract and presumably expected by API consumers, but `MySqlGetDataService.ExecuteAsync` never reads any of them — filtering/sorting/search silently do nothing. This is worse than not having the feature, because a caller believes it is filtering server-side and is not.
- Row-count is never checked after `UPDATE`/`DELETE` execution — an update against a non-existent id "succeeds" with 0 rows affected and returns success to the caller.
- No optimistic concurrency (no row version / `updated_at` compare-and-swap) — concurrent updates silently overwrite each other (lost update problem).
- `DataTypeConverter.GenerateAutoValue` uses `new Random()` (not cryptographically strong, not seeded per call in a way that avoids clustering under load) to fabricate primary-key-like values for unspecified fields — this is a data-integrity smell for anything used as an identifier.
- Single hardcoded two-value `DatabaseProvider` enum (`MySQL`, `Oracle`) throughout the codebase — no provider abstraction (no `DbProviderFactory`/strategy pattern), so adding PostgreSQL, SQL Server, or SQLite today means touching every switch statement in `DatabaseConnectionFactory`, `DatabaseConnectionVerifier`, and both `ServiceCollectionExtensions` classes.
- No caching layer anywhere (metadata lookups — `GetFieldMappersAsync`, `GetQueryDefinitionAsync` — hit the database on every single request outside the in-request `mappersCache` dictionary, which only survives one transaction).
- No resiliency policies (Polly or otherwise) around the actual data commands — only the initial connection-open has retry logic; a transient failure mid-query is not retried.
- No distributed tracing / correlation ID propagation from HTTP request → SQL execution → logs.

## Technical Debt

- Duplicate `DatabaseProvider` enum definitions (Core/Domain and Core/Enums) in different namespaces.
- Domain types physically located in `DataEngine.Core` but namespaced `DataEngine.ReaderService.*`.
- `IValidationService`/`ValidationService` is unused, unregistered code that should either be wired in or removed to avoid maintainer confusion.
- Generic `catch (Exception ex)` blocks in `DataController.FetchData` and `MySqlGetDataService.ExecuteAsync` return `ex.Message` directly to the HTTP client — an information-disclosure risk (stack traces, connection details, schema names can leak).
- `TransactionService` has no unit-test seams: it takes concrete `DbConnection`/`DbTransaction` objects and builds SQL inline in private methods, making it hard to test the SQL-generation logic without a live database.

## Scalability Risks

- Every metadata lookup (`GetFieldMappersAsync`, `GetQueryDefinitionAsync`) is a live database round-trip on every request; at high volume this multiplies database load for effectively static reference data.
- `DatabaseConnectionFactory` opens/creates a new logical `DbConnection` object per request rather than resolving from a shared, pre-configured `DbDataSource`/pool keyed by provider — relies entirely on the underlying driver's internal pool, with no visibility into pool exhaustion, and no circuit breaker to stop hammering a database that is already failing.
- `MySqlGetDataService` uses `SELECT SQL_CALC_FOUND_ROWS`, which is MySQL-specific and known to be slower than a separate `COUNT(*)` query (or a window function) on large tables — this does not scale well to large result sets.
- No caching means the metadata tables (`de_query_definitions`, `de_field_mappers`) become a hot, contended read path under load.
- No connection/command timeout configuration is visible anywhere — a slow query can hold a connection (and, in the transaction path, a row lock) indefinitely.

## Security Risks

- **P0**: SQL injection / arbitrary table access via unvalidated table names in the transaction write path (details in `CRITICAL_ISSUES.md`).
- Blocklist-based SQL firewall (`SqlGuardian`) for the direct-query read path, which is inherently bypassable and should be paired with, or replaced by, an allow-list of registered query definitions plus strict parameterization.
- Exception messages returned verbatim to HTTP clients in multiple controllers/services.
- No authentication/authorization visible on `DataController`/`TransactionController` — every endpoint is open by default (`app.UseAuthorization()` is called, but no `[Authorize]` attributes or auth scheme registration exist in `Program.cs`).
- No audit trail for who changed what — a hard requirement for most enterprise compliance regimes (SOX, ISO 27001, etc.).

## Production Risks

- Oracle provider is wired into configuration and DI but non-functional at runtime for reads and entirely absent for writes — a config mistake (`Provider: Oracle`) fails silently at startup verification only for the connection test, then fails on every subsequent read request.
- No idempotency: any client-side retry on a transaction call (timeout, load balancer retry, mobile network flake) can double-insert data with no server-side protection.
- No structured audit/rollback trail makes production incident investigation (“who deleted this row and when”) impossible today.
- Silent no-op filtering/sorting/search creates a correctness risk that will surface as a confusing support ticket (“the API isn't filtering”) rather than a clear error.

## Recommended Future Architecture

1. **Provider abstraction layer** — introduce `IDbProviderStrategy` (connection creation, parameter binding, pagination SQL, identifier quoting) implemented per provider (MySQL, PostgreSQL, SQL Server, Oracle, SQLite), selected via a factory, so `DatabaseConnectionFactory`/`TransactionService`/`GetDataService` stop branching on `switch (provider)` themselves. See `MULTI_DATABASE_DESIGN.md`.
2. **Identifier allow-list / schema catalog** — a `SchemaCatalogService` that loads the set of tables and columns a caller is permitted to touch (from `de_field_mappers`/`de_query_definitions`, cached) and validates every table/column name against it *before* any string is built, independent of the SQL text firewall. See `CRITICAL_ISSUES.md` P0-1 and `MULTI_DATABASE_DESIGN.md`.
3. **Idempotency layer** — a distributed idempotency store (Redis-first, DB fallback) keyed by `TransactionId`, returning the prior result on retry instead of re-executing. See `IDEMPOTENCY_DESIGN.md`.
4. **Audit pipeline** — an `IAuditService` invoked inside the same DB transaction as the mutation, capturing before/after images, `UserId`, `CorrelationId`, IP, and timestamp. See `AUDIT_LOGGING_DESIGN.md`.
5. **Two-tier caching (L1 in-memory + L2 Redis)** for metadata (`FieldMapper`, `QueryDefinition`) with explicit TTL and event-based invalidation. See `CACHING_DESIGN.md`.
6. **Resiliency policies (Polly)** wrapping every database call — retry with jitter for transient/deadlock errors, circuit breaker per provider/node, and command timeouts — not just the initial connection open. See `RESILIENCY_DESIGN.md`.
7. **Observability** — OpenTelemetry tracing/metrics with correlation ID propagated from HTTP → SQL → logs → audit record. See `OBSERVABILITY_DESIGN.md`.
8. **Re-enable and wire `IValidationService`** into the transaction pipeline ahead of SQL generation, or remove it — do not leave it half-built and unreferenced.
9. **Consistent namespacing** — move Core domain types into `DataEngine.Core.Domain`/`DataEngine.Core.Enums`, delete the duplicate `DatabaseProvider` enum, and have ReaderService/TransactionService reference Core rather than the reverse implied by current namespaces.
